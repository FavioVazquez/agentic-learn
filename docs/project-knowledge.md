### What this project does
An agent skill that turns any AI coding assistant into a learning partner. Instead of handing over answers, it forces productive struggle — the cognitive effort that neuroscience shows is required for real learning. Users invoke 9 structured actions from inside their agent (Windsurf, Cursor, Copilot, etc.) to learn concepts, quiz themselves, reflect, design collaboratively, journal decisions, and build a living comprehension log of their projects.

### Architecture overview
**No executable code.** The entire skill is markdown instructions that an agent reads and follows.

- `SKILL.md` — the brain. YAML frontmatter (name, description, version) + 9 action definitions with step-by-step instructions and hard constraints for the agent.
- `references/` — supporting context the agent draws on during specific actions:
  - `learning-science.md` — the neuroscience research behind the 4 techniques
  - `struggle-ladder.md` — the 4-level hint escalation system
  - `either-or-format.md` — decision journal template + 3 worked examples
- `EXAMPLES.md` — full realistic dialogues for every action (for humans, not the agent)
- `install.sh` — bash installer with `--global`, `--uninstall` flags; falls back from git to curl
- `README.md` — user-facing: install options, action table, files written

### Entry points
```bash
# Install (3 options):
npx skills add FavioVazquez/agentic-learn        # via skills CLI
curl -fsSL .../install.sh | bash                 # via curl
git clone --depth 1 ... .windsurf/skills/        # manual

# Use (inside agent):
@agentic-learning <action>
```

### The 9 actions
| Action | Core purpose |
|--------|-------------|
| `learn` | Retrieval + generation: user explains first, agent fills gaps |
| `quiz` | Retrieval practice on current file/topic |
| `reflect` | 3-part structured reflection: learned / goal / gap |
| `space` | Schedules revisits, writes `docs/revisit.md` |
| `brainstorm` | Design dialogue, writes `docs/brainstorm/YYYY-MM-DD-<topic>.md` |
| `explain-first` | User narrates code before agent comments |
| `struggle` | 3-hint ladder, user-controlled escalation |
| `either-or` | Decision journal, appends to `docs/decisions/YYYY-MM-DD-decisions.md` |
| `explain` | Reads project, writes comprehension log to `docs/project-knowledge.md` |

### Key concepts to understand
1. **The `description` field is the trigger.** Agents use it to decide when to invoke the skill automatically. Too vague = never triggers. Too broad = triggers on everything.
2. **The skill writes to the user's project, not its own directory.** All output (brainstorm docs, decisions, revisit notes, this file) goes into `docs/` in whatever repo the user is working in.
3. **`explain-first` vs `explain` are opposites.** `explain-first` = user explains TO the agent (retrieval practice). `explain` = agent explains TO the user (onboarding/comprehension).
4. **The install creates a symlink** (when using `npx skills`). Changes to the source repo are reflected immediately without reinstalling — but this also means the skill folder and the installed copy are the same object in this repo.
5. **`productive struggle` is a hard constraint, not a guideline.** The agent must not reveal answers before the user attempts them. This is the most common failure mode to watch for.

### Non-obvious things
- `skills-lock.json`, `.agents/`, and `.windsurf/` are generated by `npx skills add` and are gitignored — they should never be committed.
- The `install.sh` script installs to `.windsurf/skills/` (Windsurf-native path), while `npx skills` installs to `.agents/skills/` and symlinks to `.windsurf/skills/`. Both end up in the same Windsurf location.
- Version is currently `"1.1"` — bump it in `SKILL.md` frontmatter for any behavioral change to help users know when to update.
- The `either-or` action is named after Kierkegaard's book but is deliberately non-philosophical in execution — it's a pragmatic decision journal, not a philosophy exercise.
- `brainstorm` has a **hard gate**: the agent must not write any code or scaffold anything until the user explicitly approves the design. This is the most important constraint in that action.

### Suggested learning path for a new contributor
1. Read `SKILL.md` top to bottom — understand every action's constraints before touching anything
2. Run through `EXAMPLES.md` — see what correct agent behavior looks like
3. Install with `npx skills add FavioVazquez/agentic-learn` and test each action once
4. Read `references/` to understand the science and philosophy behind the design
5. Run `@agentic-learning explain` on any project you want to contribute to — eat your own dog food
